Detailed Comments
This paper proposes a technique for enabling Java code to interact well with host hardware devices. The claim is that the technique offers superior facilities than existing 'foreign function interfaces' (FFIs) in use in the embedded systems domain.

Section 2 is an interesting overview of areas in which different FFIs offer different capabilities. In terms of 'direct hardware access' it is pointed out that the standard JNI relies on dynamic class loading and so is inappropriate in embedded environments. But it is made clear that the JME's FFI is also deficient in not allowing the creation of new objects at runtime, in not allowing 2-way interaction between the hardware to Java, and in only allowing access to calls that are pre-built into the JVM. Section 2 also explores design choice between JIT compilation and 'ahead-of-time' compilation (i.e. what is normally known as just 'compilation' :-)). It is concluded that the latter is a good approach for Java-based embedded systems. It also clearly makes it easier to interact flexibly between 'Java' (compiled to C!) and C in the low level foreign environment (e.g. avoiding being restricted to calls pre-built into a JVM and easily supporting 2-way interaction). It is clear by the end of section 2 that the proposed solution is going to claim to avoid some of the common state of the art limitations by adopting an 'ahead-of-time' compilation approach which indeed facilitates 2-way interactions and flexibility beyond calling JVM built-ins (although it will presumably not avoid the 'creation of new objects at run time' limitation that JME was criticized for).

The paper then delves into the details of the proposed solution. These seem unfortunately very specific to the authors' particular environment consisting of the KESO VM and the EPOS OS. Furthermore, some of the explanations of the JVM and OS are pretty obscure (especially the key stuff at the top of page 6) which makes it hard to follow all the detail later on.

The proposal itself, once the VM and OS have been introduced, is quite straightforward and only takes half a page or so to explain (p7/8). The proposal is then illustrated with a case study of a Java program that interfaces to a UART. This is OK as far as it goes, but I would like also to have seen a definitive and formal statement of exactly what are the Java-related semantics of the proposed FFI. For example, can it support object instantiation? exceptions? how does GC work with it? It would also have been interesting to have seen a discussion of how specific the proposal is to KESO and EPOS. It would clearly be more useful if it could generalize to other environments.

The evaluation is disappointing: it is simply shown that the executable images for Java versions of the case study program are larger than those for native C++ versions. But it was always pretty clear that this was going to be the case. Unfortunately, there is no baseline or context offered and therefore no basis on which we can interpret these results (are they good or bad? very good? why?).

It should also be said that the paper is a bit out of scope for Middleware - there is no distributed systems content. The paper might be better targeted at a programming language/ embedded systems event.

Small points:

- Should spell out EPOS in the abstract
- Should only spell out acronymns once, not several times
- English is shoddy in places making the paper harder to follow than it should be

Negative Aspects of the Paper (please be brief) This paper lacks a clear contribution and an intuitive presentation of the technical aspects related to the C / Java binding.
Detailed Comments The paper discusses the approach taken in KESO (a toolchain to run Java programs on embedded devices) to executive native code.

I see several major issues for which this paper should be rejected. First, the motivation for this work and the advancement over the state of the art are not properly discussed. In particular, I do not understand why we need another another method to use native code in Java applications. Why is KESO FFI any better than KNI or than the native code invocation in Darjeeling (Middleware'08, SenSys'09)? Why is Java the right language for the embedded domain? I believe that these aspects should be clarified by the authors.

Second, I believe that the paper suffers from a presentation issue. There are many details that are left out of the paper or put forward in a way that is not easy to comprehend. Indeed, the paper should give enough background on KESO so that the reader gets a clear feeling of the inner workings, but I was not able to fully grasp all the technical details related to the native code invocation. For instance, I do not understand at all the first paragraph on pg. 6. What is the Wavelet class? Why do we need to extend it? How do we specify the join point in the constructor of the WaveletImplementation? Why do we overwrite the ignoreMethodBody and we do not use annotations for instance -- is this function called by the compiler? How do we register this class in an "appropriated class of the KESO compiler"? What are the parameter types converted from Java to C? The example given in Sec. 5.1 should come earlier and be used to properly discuss the API to use native code and the pertinent aspects of KESO.

Then, I believe the evaluation criteria are not properly set, i.e. why compare the Java implementation of an ENTIRE application against a native implementation of the same app, when the objective of the paper is to discuss native code invocation? IMHO, an approach similar to the following might be more appropriate: take some functionality and implement it in Java and in native code and then compare the invoking these two functionalities from KESO.

Finally, there are some terminology / aspects / statements in the paper that I do not agree with:
- "The Java class loaders are used to search for the native method and to add them into JVM". As far as I know, class loaders are used to load Java class bytecode into the JVM, and not to load native code.
- "As major hardware devices are mapped in memory addresses, direct access to them is a problem to Java language". This is a picky comment, but the purpose of the Java language is to provide an abstraction to access the hardware devices. Handling of the memory addresses or hardware access is an issue of the JVM.
- "KNI forbids creating new Java objects (other than strings), and forbis calling Java methods from native code". Again I am picky, but at the first read I understood that there is no way to generally instantiate new objects in KNI, which is not true: it is only the native code that does not have the ability to create new classes or instantiate objects.
- According to "The Java Virtual Machine Specification 2nd Ed." by Lindholm and Yellin, there is a clear distinction between the Java language and a Java virtual machine. The latter has its own instruction set and runs the bytecode contained in class files. Instead, KESO stands out as a toolchain that takes bytecode as input and generates C code that eventually is compiled in native code. I strongly disagree with the authors when they claim that "KESO is a JVM in the sense that it runs Java programs", as a JVM is much more than that.

Due to the many typos and errors, I would also recommend the authors to ask the help of an English native for proofreading the text.

Overall Rating Reject
Positive Aspects of the Paper (please be brief) This paper targets the important problem of creating an interface between Java and hardware objects such as sensors in embedded systems. The authors state that there are no suitable techniques to do this currently.
Negative Aspects of the Paper (please be brief) The paper does not make clear the novelty of this work, or the contributions that the authors have made. The explanation is sketchy and the evaluation needs to be more thoroughly performed and explained.
Detailed Comments This paper describes a method of interfacing between embedded Java and hardware such as sensors and other hardware devices.

The contributions of this paper should be outlined in one of the earlier sections. Currently it is very unclear what work has been done before and what this paper is contributing. Also, the paper needs a Motivation section laying out why the EPOS integration is essential. In the present version, the motivation is spread through the different sections of the paper.

What would also help drive the point home is a comparison with state-of-the-art techniques (such as reference 15). Currently there is no way of estimating how much better the authors' proposed technique is.

Section 4 is the main meat of the paper and should be further flushed out.
